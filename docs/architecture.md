# Homelab architecture

This document describes the high‑level architecture of the homelab, with a focus on routing ownership, control‑plane vs data‑plane, and WireGuard surfaces.

---

# Ownership model

# Router

Responsibilities:

- Owns WAN uplink
- Owns NAT
- Owns routing
- Owns firewall
- Owns public WireGuard endpoint
- Runs WireGuard interface wgs1
- Loads config generated by NAS
- Uses wg + ip for bring‑up

The router is a runtime node, not a config generator.

# NAS

Responsibilities:

- Intent storage
  - hosts.tsv
  - wg-interfaces.tsv
  - clients.tsv

- Key management
  - keys/servers/<iface>.key/.pub
  - keys/clients/<name>.key/.pub

- Config generation
  - output/router/<iface>.conf
  - output/server/<iface>.conf
  - output/clients/<name>.conf
  - output/clients/<name>.qr.txt

- Firewall script generation
  - wg7-firewall.sh (and others if needed)

- Deployment orchestration
  - Pushes router configs via SSH + atomic installer
  - Installs NAS configs locally

- Bring‑up orchestration
  - Router: wg + ip
  - NAS: wg-quick + firewall script

The NAS is the control plane node: it owns intent, compilation, and deployment, but not routing.

# Clients

- Consume configs generated by NAS
- Connect to router’s public WireGuard endpoint (wgs1)
- Do not own any control‑plane logic

# WireGuard surfaces

# Router

- Interface: wgs1
- Role: Public VPN entry point
- Config source: output/router/wgs1.conf
- Bring‑up: wg + ip

# NAS

- Interfaces (enabled in TSV):
  - wg1–wg7, wg14, wg15
- Primary used interface: wg7
- Role: Internal VPN surface for NAS services
- Bring‑up: wg-quick up wg7 + wg7-firewall.sh

Other enabled NAS interfaces are generated but only become active if deployed.

# Control plane vs data plane

# Control plane (NAS)

- Reads TSVs
- Generates configs
- Generates firewall scripts
- Manages keys
- Deploys configs
- Orchestrates bring‑up

# Data plane (Router + NAS + Clients)

- Router forwards packets between VPN, LAN, WAN
- Router applies firewall and NAT
- NAS terminates wg7
- Clients send/receive traffic via router’s wgs1

# Deployment pipeline (WireGuard)

1. Intent definition (TSVs)
   - hosts.tsv
   - wg-interfaces.tsv
   - clients.tsv

2. Generation (NAS)
   - bin/generate-configs.sh
   - bin/generate-firewall.sh

3. Deployment
   - Router: /jffs/etc/wireguard/wgs1.conf
   - NAS: /etc/wireguard/wg7.conf + firewall script

4. Bring‑up
   - Router: create interface, load config, set link up
   - NAS: wg-quick up wg7 + firewall script

# Invariants

- Router remains the authoritative routing and NAT node
- NAS is the authoritative config and key generator, not a routing authority
- All WireGuard configs are generated from TSVs
- Deployment is atomic and idempotent
- Disabled interfaces in TSV are ignored
- Clients always connect to router’s public endpoint (bardi.ch:<port>)
